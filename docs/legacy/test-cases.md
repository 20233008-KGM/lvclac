## 1. 핵심 계산식

### 1.1 공통 — 1포인트당 손익이 출발점

청산가 계산의 핵심은 **가격 1포인트(1단위)가 움직일 때의 손익 크기**다.

```
총 민감도 Q = 보유 계약수(N) × 계약승수(M)      ← 1포인트당 손익
약정가치(명목) = N × 1계약 약정금액 × M          (약정금액·승수가 있을 때)
포인트가치 = 1계약 약정금액 × M ÷ 현재가
```

가격이 `ΔP` 움직이면 손익 = `ΔP × Q` (롱은 +, 숏은 −).

### 1.2 증거금 — 입력 방식별 (이 문서의 핵심 분기)

| 입력 방식 | `marginInputMode` | 유지/개시 증거금 산식 | 가격 변동 시 |
|-----------|-------------------|------------------------|--------------|
| **비율** | `rate` | `명목 × 증거금률` | 가격에 **비례** (명목이 가격 따라 변함) |
| **계약당 고정** | `perContract` | `계약당 고정금액 × 계약수` | **고정** (가격과 무관, 해외선물) |
| **총액** | `total` | HTS 표시 총액 (계약수 비례 조정) | 가격에 **비례** |

```
1계약당 유지증거금 = 포지션 유지증거금 ÷ 계약수
1계약당 개시증거금 = 포지션 개시증거금 ÷ 계약수
가용증거금 = 계좌평가금액 − 개시(위탁)증거금
유지증거금 여유 = 계좌평가금액 − 유지증거금
```

### 1.3 청산가 — 유지증거금이 가격에 비례하는지로 분기

현재가 `C0`, 계좌평가금액 `E0`, 총 민감도 `Q`, 현재가 기준 유지증거금 `M0`.

**(A) 비율 / 총액 — 유지증거금이 가격에 비례** `M(P) = M0 × P/C0`

```
롱 청산가 = (C0×Q − E0) ÷ (Q − M0/C0)
숏 청산가 = (E0 + C0×Q) ÷ (Q + M0/C0)
(비율 입력이면 M0 = C0×Q×R 이므로:
 롱 = (C0×Q − E0) ÷ (Q×(1−R)),  숏 = (E0 + C0×Q) ÷ (Q×(1+R)))
```

**(B) 계약당 고정 — 유지증거금이 상수** `M(P) = Mfix`

```
롱 청산가 = C0 + (Mfix − E0) ÷ Q
숏 청산가 = C0 + (E0 − Mfix) ÷ Q
```

> 청산가는 `청산가 > 0` 이고 방향이 맞을 때만 표시한다(롱은 현재가 아래, 숏은 현재가 위). 그 외에는 `-`.

### 1.4 청산 여유 · 레버리지 · 추가 주문

```
롱 여유(%) = (C0 − 청산가) ÷ C0 × 100
숏 여유(%) = (청산가 − C0) ÷ C0 × 100
청산까지 변동폭 = 롱: C0 − 청산가 / 숏: 청산가 − C0
레버리지 = 약정가치 ÷ 계좌평가금액
추가 매수/매도 한도 = floor((계좌평가금액 − 개시증거금) ÷ 1계약당 개시증거금)
주문 체결 손익 = 롱: (C0 − 주문가) × Qord / 숏: (주문가 − C0) × Qord   (Qord = 주문계약수 × 승수)
```

### 1.5 시나리오 모드

```
시나리오 평가금액 = 진입 시 계좌평가금액 + (시나리오가 − 현재가) × Q   (롱 +, 숏 −)
```

- Enter 1회 → 시나리오 미리보기 진입 (현재가·계좌평가금액 입력값은 유지, 결과만 시나리오가 기준)
- 시나리오 모드 중 Enter → 손익 반영(현재가·계좌평가금액 롤링)
- Esc → 시나리오 모드 종료, 진입 전 상태 복원
- Ctrl+Z → 손익 반영 취소

#### 일상 사용 — 매일 시세 반영

포지션·증거금이 그대로일 때, 계좌 평가금액을 매번 HTS에서 다시 입력하지 않고 시나리오 가격으로 갱신할 수 있다.

1. 비로그인은 「이 기기에 입력값 저장」, 로그인 사용자는 「이 기기」 또는 「클라우드」 저장 위치 선택 후 어제 상태 유지
2. HTS 시세를 시나리오 가격에 입력 → Enter (미리보기)
3. HTS와 비교 후 Enter (손익 반영)

적용 조건: 계약수·종목 불변, 입출금·수수료·실현손익 없음, 가격 변동만 반영.

**입력 타이밍:** 평가금액과 현재가는 동일 시점 기준이 이상적이다. HTS는 한 화면 스냅샷, MTS는 잔고·시세 화면 전환이 필요해 간극이 더 클 수 있다. 변동성이 큰 구간에서는 입력 순서 사이 미세 오차 가능 → 변동성 낮을 때 기입, 또는 캡처 스냅샷 후 기입 권장. 최초 세팅 시 현재가를 먼저 넣고 바로 이어서 평가금액을 넣는 순서가 유리하다.

## 3. 정답값 기준 데이터셋 (모드별)

각 모드의 대표 입력과, 그 입력으로 산출되는 롱/숏 청산가를 테스트로 고정한다.

### 3.1 비율(`rate`) — 🇰🇷 KOSPI200 (검증: `liquidation.test.ts`)

| 입력 | 값 |
|------|----|
| 계좌평가금액 | 73,511,744 |
| 유지/개시 증거금률 | 0.247 / 0.35 |
| 보유 계약수 | 58 |
| 계약승수 | 10 → **Q = 580** |
| 현재가 | 320,500 |

| 결과 | 롱 | 숏 |
|------|----|----|
| 청산가 | ≈ **257,312** | ≈ **358,656** |
| 청산 여유(%) | ≈ 19.72% | ≈ 11.91% |

### 3.2 계약당 고정(`perContract`) — 🌎 해외선물 (검증: `perContractMargin.test.ts`)

| 입력 | 값 |
|------|----|
| 계좌평가금액 | 10,000 |
| 유지/개시 (계약당) | 20 / 50 |
| 보유 계약수 | 100 |
| 계약승수 | 1 → **Q = 100** |
| 현재가 | 100 |

산출: 유지증거금 = 20×100 = **2,000(고정)**, 개시 = 50×100 = **5,000**

| 결과 | 롱 | 숏 |
|------|----|----|
| 청산가 | C0 + (2000−10000)/100 = **20** | C0 + (10000−2000)/100 = **180** |
| 추가 매수/매도 한도 | floor((10000−5000)/50) = **100** | 동일 |

> 핵심: 유지증거금 2,000은 가격이 20이든 180이든 **변하지 않는다**. 그래서 비례식이 아니라 고정식을 쓴다.

### 3.3 총액(`total`) — HTS 표시 총액 (검증: `perContractMargin.test.ts`)

| 입력 | 값 |
|------|----|
| 계좌평가금액 | 3,600 |
| 유지/개시 (총액) | 2,000 / 3,000 |
| 보유 계약수 | 2 |
| 계약승수 | 1 → **Q = 2** |
| 현재가 | 5,000 |

| 결과 | 롱 | 숏 |
|------|----|----|
| 청산가 | (5000×2−3600)/(2−2000/5000) = **4,000** | (3600+5000×2)/(2+2000/5000) ≈ **5,666.67** |

> 총액 모드의 유지증거금은 가격에 비례한다고 가정한다(국내 HTS 총액 = 비율 산출값의 스냅샷). 해외 고정 증거금은 `계약당` 모드를 사용할 것.

## 5. 회귀 · 엣지 체크리스트

- [ ] 모드 전환 시 각 모드 입력값이 보존된다(별도 필드 저장).
- [ ] 구버전 저장값(모드 없음) 복원: 직접 총액 있으면 `total`, 없으면 `rate`로 마이그레이션.
- [ ] `유지 > 개시`이면 경고(`maintenance_rate_exceeds_entrusted`) — 같은 모드일 때만 비교.
- [ ] 계약당/총액 모드에서 약정금액 없이도 청산가 산출(현재가+계약수+증거금만으로).
- [ ] 계약당 모드: 가격을 크게 흔들어도 유지증거금이 변하지 않음(고정식).
- [ ] 시나리오 모드 진입 중에는 증거금 모드 토글·입력 비활성화.
- [ ] 추가 한도 초과 주문 시 `order_exceeds_max_buyable` / `order_exceeds_max_sellable`.
